# 2.2 CUDA C Program Structure

When writing a CUDA C program, the goal is to exploit data parallelism by writing code that can run simultaneously across thousands of cores.

To achieve this, the program is split into two parts:

1. **Host Code (CPU):** This is traditional, sequential C code that runs on the CPU. It handles setup, data loading, and orchestrating the GPU.
2. **Device Code (GPU):** This is the parallel code that runs on the GPU. The functions executed on the GPU are called **kernels**.

When the CPU calls a kernel function, it tells the GPU to spawn a massive number of threads. All the threads launched by a single kernel call are collectively called a **Grid**. When the grid finishes executing, control returns to the CPU.

## The Cost of Threads: CPU vs GPU

A major conceptual hurdle in parallel programming is understanding why we can effortlessly launch millions of threads on a GPU, but attempting to launch even 1,000 threads on a CPU will crash the system or bring it to a crawl.

This difference comes down to two hardware mechanisms: **Generating** and **Scheduling**.

### What does "Generating" a thread mean?

Generating a thread means physically creating it in the system memory so it can begin executing.

* **On a CPU:** Creating a thread requires an expensive Operating System software call. The OS has to stop what it's doing, allocate a massive chunk of RAM for the thread's stack (often 1MB to 8MB), set up security contexts, and register the thread in the system's software scheduler queue. This process takes **thousands of clock cycles**. It is incredibly slow and heavy.
* **On a GPU:** The GPU hardware is explicitly designed to create threads directly in silicon. Generating a GPU thread requires zero Operating System involvement. The hardware simply increments an internal counter and maps the thread to pre-existing hardware registers. This process takes **1 or 2 clock cycles**. It is practically instant.

### What does "Scheduling" a thread mean?

Scheduling means deciding exactly *when* an active thread gets to physically use the processor to execute an instruction, and efficiently pausing threads that are waiting.

* **On a CPU:** When the CPU switches from running Thread A to running Thread B, the OS must perform a "Context Switch." It has to take all of Thread A's current variables, save them out to the RAM, and then load Thread B's variables from the RAM into the CPU registers. Moving data back and forth to RAM is extremely slow, making CPU scheduling very expensive.
* **On a GPU:** A GPU Streaming Multiprocessor (SM) has a massive physical register file. When a block of threads is assigned to an SM, *all* of their variables are stored permanently in the hardware registers for the entire lifetime of the thread. If Thread A needs to pause (perhaps waiting for memory), the hardware scheduler instantly switches to Thread B. There is no context switch to RAM because Thread B's data is already sitting perfectly still in the registers, ready to go. This swap takes **zero extra clock cycles**.

Because GPU threads are so lightweight to generate and free to schedule, we can launch millions of them without overwhelming the hardware.

## Breaking Down "Threads"

To truly understand how a massive Grid operates, we must first deeply understand what a single thread actually is at the hardware level.

**1. A thread is a simplified view of how a processor executes a sequential program.**
A thread is simply a single, independent worker. It looks at a list of instructions and executes them one by one, from top to bottom, in exact order.

**2. A thread consists of the code, the execution point, and its variables.**
For a thread to exist, it only requires three pieces of state:

* **The Code:** The list of compiled instructions in memory.
* **The Program Counter:** A hardware bookmark pointing to the exact line of code the thread is currently reading.
* **The State:** The actual values stored in its local variables and hardware registers at this exact millisecond.

**3. The execution of a thread is strictly sequential.**
A single thread never performs two actions simultaneously. It will completely finish executing line 1 before it begins executing line 2. The parallelism in a GPU does not come from a single thread doing multiple things; it comes from having millions of these sequential threads running side-by-side.

**4. A thread's progress can be monitored statement by statement.**
Because a thread is completely sequential and strictly owns its own local variables, its execution is completely deterministic. If you attach a debugger to a single thread, you can freeze it in time. While frozen, you can inspect the Program Counter to see what instruction will happen next, and you can read the registers to see exactly what numbers the thread is currently holding.

**5. "Multi-threading" usually refers to traditional CPU parallelism.**
Technically, any program running more than one thread is "multi-threading." However, in the software industry, when people say "multi-threading," they are usually talking about **CPU Task Parallelism**. For example, a developer writes code to manually spawn 4 heavy OS threads: Thread 1 runs the UI, Thread 2 handles networking, Thread 3 saves to the disk, and Thread 4 plays audio. Each thread is assigned a completely different function.

**6. CUDA uses "Massively Parallel SIMT", not traditional multi-threading.**
While a GPU *does* use multiple threads, we generally don't call it "multi-threading"; we call it **Massively Parallel** (specifically SIMT: Single Instruction, Multiple Threads). Instead of manually spawning 4 threads to do 4 different tasks, a CUDA program calls a single kernel function. This triggers the hardware to instantly launch a "Grid" of millions of threads. Every single thread in the Grid executes the **exact same sequential code**, but uses its unique Thread ID to pull a **different piece of data** from memory. This achieves massive Data Parallelism, which is a completely different beast than CPU multi-threading.
